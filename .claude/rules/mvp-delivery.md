# MVP Delivery Rules

## Current Mode: MVP Delivery

## Objective
Deliver the smallest working onboarding-to-first-transaction flow for the Multi-Industry POS Platform.

## Definition of Done
- [ ] User can sign up / login
- [ ] User can create business profile
- [ ] User can create first branch
- [ ] User can add first product
- [ ] User can invite first staff
- [ ] User can complete first transaction
- [ ] Basic RBAC is enforced
- [ ] Flow works on tablet-sized screen

## Rules
- Prioritize working functionality over completeness
- Avoid introducing new abstractions unless necessary
- Do not optimize prematurely
- Focus only on enabling the first successful transaction
- Prioritize fixing gaps over creating new features
- Do not introduce Phase 3+ features into MVP work
- Do not redesign approved architecture without strong justification

## Out of Scope
- Mobile app implementation
- Advanced analytics
- Deep reporting
- HR module
- Contacts module
- UI polish beyond usability
- Enterprise ERP-style features

## Agent Behavior Under MVP Mode
- Complete only the assigned task — do not expand scope
- If a task touches something out of scope, flag it and stop
- Prefer the simplest solution that makes the feature work
- Do not refactor unrelated code while implementing a fix
- Report blockers immediately rather than working around them silently
